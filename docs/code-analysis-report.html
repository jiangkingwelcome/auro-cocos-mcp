⚠️ 当前遗留问题

P1 — 建议优先处理

1
core.ts 仍然 843 行，路由定义占大量篇幅
建议将 API 路由拆分为 routes-api.ts、routes-asset.ts 等子模块，core.ts 仅保留服务器启动和安全层。
2
tools-editor.ts editor_action 有 38 个 case 分支
圈复杂度极高。建议参照 scene.ts 的 Map 策略模式重构，或按功能拆分为 editor-scene、editor-panel、editor-build 等子处理器。

P2 — 可后续处理

3
core.ts / scene.ts 缺少专属单元测试
HTTP 路由逻辑和场景脚本的 Map handler 缺少直接测试。建议为纯逻辑部分（安全检查、路由匹配）编写可独立测试的函数。
4
tools-atomic.ts 中 2 个 stub 工具
create_tween_animation_atomic 和 auto_fit_physics_collider 返回 success: false。建议实现或标注为 @experimental。
5
local-tool-server.ts 存在两套 Zod→JSON Schema 转换
zodToJsonSchemaProp (手写) 与 tools-shared.ts 的 toInputSchema (使用 z.toJSONSchema) 功能重复。建议统一。
6
tools-shared.ts 中 CASE_NORMALIZE_MAP 为空对象
normalizeDbUrl 实际上永远不会做任何规范化。应移除死代码或实现功能。
7
panels/default/index.ts 使用 @ts-nocheck
整个面板文件跳过类型检查（747 行）。受限于 Cocos Panel API 的 webview 约束，短期内难以改善。
8
editor.d.ts 中 Function 类型过于宽泛
PanelDefineOptions.methods 使用 Record<string, Function>。建议改为更具体的签名。

★
零 any 类型安全

★
10 层纵深安全防护
从网络层（回环限制）到应用层（Token/速率限制）到业务层（IPC 白名单/属性黑名单/危险操作确认），形成完整的安全纵深。
★
原子宏的阶段追踪 + 自动回滚
create_prefab_atomic 等原子工具实现了 8 阶段追踪，任一阶段失败自动回滚已完成的操作，并在响应中报告每个阶段的状态。
★
AI 友好的 Guardrail 系统
组件名自动纠正（sprite→Sprite）、路径规范化、错误时附带建议（"可尝试组件名: ..."）、危险操作需要 confirm: true。让 AI 客户端更容易正确使用。
★
stdio-shim 的健壮重连机制
自动发现 MCP Host、帧格式自动检测（JSON-Lines/Content-Length）、断线自动重连 + 补发 initialize、项目切换检测 + 警告注入、日志轮转（2MB 限制）。
★
模块化工具注册架构
